Large - pwn

先给结论：这题我已经跑通利用链了，脚本放在 large/exp.py。

题目结论

  • 保护：Full RELRO / Canary / NX / PIE / SHSTK / IBT
  • 漏洞本质：UAF
    • delete 后不清空 chunks[idx]
    • 所以还能 show / edit 已释放 chunk
  • 尺寸限制：0x501 ~ 0x5ff
    • 这让 chunk 直接走 unsorted/largebin，基本就是在暗示 largebin attack

利用链

  1. gift 调用内置的 printf("%p\n", &g_f)
    → 直接泄露 PIE（准确说是 g_f 的地址）

  2. show 已释放的大块
    → 读出 unsorted bin 的 fd/bk
    → 泄露 libc

    • 我本地算出的偏移是：main_arena+0x60 = libc + 0x203b20

  3. 先把 chunk1 放进 largebin

  4. 用 UAF edit chunk1 改它的 bk_nextsize = g_f - 0x20

  5. 再把 chunk3 放进同一 largebin
    → 触发 largebin attack
    → 把 g_f 改成 chunk3 的 chunk header

  6. 再 UAF 改 chunk3 内容，伪造成：

    • system
    • "/bin/sh"
    • 0
  7. 再次 gift
    → 实际变成 system("/bin/sh")

ONNX 模型与“衔尾蛇 (Ouroboros)”的关系

在计算机科学和机器学习的语境下，ONNX（Open Neural Network Exchange） 是用于表示深度学习模型的标准格式。从数据结构上讲，标准的 ONNX 模型必须是一个 DAG（有向无环图，Directed Acyclic Graph），这意味着它的数据流是单向的，绝对不允许存在真正的物理环路。

然而，“衔尾蛇 (Ouroboros)” 的核心隐喻是“吞噬自己尾巴的蛇”，象征着无限循环、自我递归或首尾相连。

这两者结合在一起，指向了这道题的核心考点——这是一个 ONNX Quine（自产生程序）或者在元数据层面首尾相连的“伪环”模型。

  1. 结构上的首尾相连：仔细看你提供的文本末尾，在执行完所有计算到达 dead_add_2519_ee66eb 后，它通过一个自定义属性 onnx_bit_chain 直接将尾部强行关联回了最初的输入节点 answer_weight_0001_8f79b3。它在 DAG 的限制下，用元数据画出了一个逻辑上的圆（衔尾蛇）。
  2. 零输入与自生成：模型图内部充斥着大量的常量节点（如 add_zero、mul_one），这意味着它不需要外界提供任何输入（Input），仅凭内部的张量进行算术逻辑推演，最终输出自身的网络结构代码或隐蔽的 Flag（即输出 answer_codes）。

尝试还原 Flag 的思路

从你提供的 strings 层面的文本，我们可以看出这道题有两种解法（静态分析提取与动态执行还原）：

方法一：静态分析提取比特链（Bit-chain）

文本中存在大量的数学上的无用操作（NOP）：

  • add_zero (加 0，值不变)
  • mul_one (乘 1，值不变)

出题人故意使用这些冗余操作，实际上是为了在计算图中嵌入一个二进制流：

  • 所有的 add_zero 节点代表二进制的 0
  • 所有的 mul_one 节点代表二进制的 1

你可以按节点的索引顺序（如 0010、0013、0031…）将这些特征提取出来。例如最开始的几个特征：
add_zero_0010 (0), mul_one_0013 (1), mul_one_0031 (1), mul_one_0034 (1), mul_one_0072 (1), add_zero_0109 (0), mul_one_0133 (1), mul_one_0146 (1)
这前 8 个操作组成的二进制是 01111011，其对应的十进制是 123，正好对应 ASCII 字符中的 { ！（Flag 格式的标志性字符）。
如果将所有这些节点以及末尾处于“dead”状态的 dead_zero 收集全并按每 8 位转换为 ASCII，大概率能直接静态解出被切碎的 Flag。

方法二：动态执行模型（推荐，最快捷）

既然这是一个自驱动的模型，它最终会把所有的二进制链进行拼接、重塑、切片处理，并输出给最终的 answer_codes。

完整攻击链路总结

1. 信息收集

  • 登录 demo/demo123，发现 /api/workflows、/api/nodes、/webhooks/:id、/api/nodes/import 等端点。
  • 在 /workflows/demo-workflow 的构建日志中发现唯一一条真实日志，指向 /support/incidents/FL0WF0RG3-SG2026-PR3V13W-C0MP47。
  • 下载工单页面里的 artifact 包 flowforge-partial-src.zip，里面是部分服务端源码、admin_graphql.txt、mcp_inspector.txt、deployment_notes.txt，揭示了关键的内部链：
    • 127.0.0.1:9001/graphql 用 x-internal-token（runner token）调用 debugExport(flowId) 拿到 exportToken
    • 127.0.0.1:9100/debug/export_secret_bundle 用 x-mcp-export-token 兑换 secret bundle

2. Webhook 签名不一致漏洞（核心 0day）

/webhooks/:workflowId 用 firstKeyJsonParse(rawBody) 验签，但用标准 JSON.parse(rawBody) 处理业务事件：

构造重复键 body（demo workflow 的 webhook secret whsec_demo_public_001 是公开的）：

  • firstKeyJsonParse → event=workflow.ping（绕过 actor role 检查）
  • JSON.parse → event=workflow.promote（实际执行 promote 逻辑）

服务器返回：{"status":"ok","message":"User promoted","runner_token":"runn3r_fl0wf0rg3_..."}，账户升级为 maintainer 并获得 runner token。

3. MCP 节点注入 → RCE

作为 maintainer 调用 POST /api/nodes/import 注入一个恶意 processor 节点，config.script 里写 Node.js 代码（链式访问 GraphQL + MCP inspector）。

4. 触发构建获取 secret bundle

POST /api/builds/trigger 指定 workflow_id=demo-workflow（让 getLatestRunnerTokenForUser 能取到刚才 promote 拿到的 runner token），node_id=3，runner 在内网执行：

  1. POST 127.0.0.1:9001/graphql 带 x-internal-token: <RUNNER_TOKEN> 调用 debugExport(flowId:"admin-secret") → 拿到 exportToken
  2. POST 127.0.0.1:9100/debug/export_secret_bundle 带 x-mcp-export-token: <exportToken> 和 {"workflow_id":"admin-secret"} → 返回 bundle，包含 flag。
